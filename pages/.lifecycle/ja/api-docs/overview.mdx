# API

<Callout type="info">
現在の記事はFeatBit REST APIへのリクエストの作り方の一般的な紹介です。APIの完全なリストやそれぞれの使用方法については、[こちら](https://featbit-tio-eu-api.azurewebsites.net/docs/index.html)にRedocで生成されたドキュメントが利用できます。FeatBitをローカルやサーバーで実行している場合は、次のURLにポイントすることもできます：<strong>URL\_OF\_API\_SERVER/docs.</strong> たとえば、FeatBitをhttpsなしでローカルマシンで実行している場合は、次のようなURLになります：<strong>http://localhost:5000/docs</strong> 。
</Callout>

### 認証

すべてのREST APIリソースは、[個人アクセス・サービスアクセストークン](../integrations/api-access-tokens.md)のいずれかで認証されます。他の認証メカニズムはサポートされていません。個人アクセス・トークンは、「統合/アクセス・トークン」ページで管理できます。

#### リクエスト・ヘッダーを使用した認証

APIとの認証の唯一の方法は、要求に含まれる`Authorization`ヘッダーにアクセストークンを含めることです。`Authorization`ヘッダーの値はアクセストークンである必要があります。このヘッダーの追加方法については、[FeatBit REST APIの使用](using-featbit-rest-api.md)を参照してください。

### 表現

すべてのリソースはJSONレスポンスボディの受け取りと返送を想定しています。エラーレスポンスもJSONボディを送信します。APIのエラー形式について詳しくは、[Errors](overview.md#errors)を読んでください。

これはつまり、常に`application/json`として設定された`Content-Type`ヘッダーのレスポンスを受け取ることになります。

さらに、`PATCH`、`POST`、`PUT`のリクエストボディは、`application/json`としてエンコードする必要があります。

### エラー

APIは常に共通のフォーマットでエラーを返します。以下はその例です：

```json
{
    "success": false,
    "errors": [
        "Unauthorized"
    ],
    "data": null
}
```

**errors**は一般的なエラークラスを示しています。

### CORS

FeatBit APIは、任意のオリジンからのAJAXリクエストのためのCross Origin Resource Sharing（CORS）をサポートしています。リクエストに`Origin`ヘッダーが含まれている場合、明示的に許可されたオリジンとしてエコーされます。それ以外の場合は、ワイルドカード`Access-Control-Allow-Origin: *`がリクエストに対して返されます。CORSに関する詳細は、[CORS W3C Recommendation](http://www.w3.org/TR/cors)を参照してください。例えば、CORSヘッダーは次のようなものです：

```
Access-Control-Allow-Headers: Accept, Content-Type, Content-Length, Accept-Encoding, Authorization
Access-Control-Allow-Methods: OPTIONS, GET, DELETE, PATCH
Access-Control-Allow-Origin: *
Access-Control-Max-Age: 300
```

認証付きのCORSリクエストは、同オリジンリクエストと同様に行うことができます。アクセストークン認証を使用して認証されたCORS呼び出しを、信頼されていないエンティティに公開してはなりません。
